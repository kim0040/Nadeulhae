# 프로젝트 계획서: 전주 나들해 (Jeonju Nadeulhae) - 날씨 기반 피크닉 & 반나절 코스 추천 서비스

## 1. 프로젝트 개요

본 프로젝트는 전주 지역의 과거 기상 데이터와 실시간 날씨 예보 데이터를 데이터베이스(DB)에 수집 및 분석하여, 사용자에게 직관적이고 예쁜 UI로 '오늘 피크닉 가기 좋은지'를 알려주는 웹 서비스입니다.

단순한 날씨 정보 제공을 넘어, 시간대별 날씨 변화에 맞춘 **'AI 기반 반나절 외출 코스 자동 생성'** 기능을 핵심 차별화 포인트로 삼아, 사용자의 완벽한 하루를 설계해 주는 웹 서비스 **'전주 나들해 (Jeonju Nadeulhae)'**를 구축하는 것을 목표로 합니다.

## 2. 기획 배경 및 차별성

- **기존 서비스의 한계:** 기존 날씨 앱은 수치(온도, 습도 등)만 제공하며, 장소 추천 앱은 날씨 변화를 실시간으로 반영하지 못합니다.
    
- **차별화 포인트:** "오후 2시엔 맑지만, 5시엔 바람이 많이 분다"는 시간대별 기상 예보 흐름을 파악하여, **"야외 활동 -> 실내 활동"으로 이어지는 자연스러운 동선(반나절 코스)**을 LLM(대규모 언어 모델)과 DB를 활용해 큐레이션 해줍니다.
    

## 3. 핵심 기능 상세 설명 (데이터 및 사용자 흐름 중심)

이해를 돕기 위해 데이터가 어떻게 흘러가고, 사용자가 어떤 흐름으로 서비스를 경험하게 되는지 상세히 풀어냅니다.

### 핵심 기능 1: 과거 데이터 통계 및 '오늘의 피크닉 지수' (기본 기능)

- **데이터 수집 흐름:** 1. 기상청 API를 통해 과거 수년간의 전주 날씨 데이터(기온, 강수량, 미세먼지 등)를 수집하여 MySQL 등의 RDBMS에 적재합니다. (배치 작업)
    
    2. 실시간 날씨 API를 주기적으로 호출하여 오늘의 예보 데이터를 DB에 업데이트합니다.
    
- **사용자 경험 흐름:** 사용자가 메인 페이지에 접속하면, 예쁜 그래픽 UI(예: 햇살이 비치는 세병호 일러스트 등)와 함께 "오늘은 피크닉 가기 완벽한 날이에요! (피크닉 지수 95점)"라는 직관적인 결과를 봅니다.
    
- **통계 제공 흐름:** 화면 스크롤을 내리면, DB에 저장된 과거 데이터를 바탕으로 "최근 3년간 5월 중 가장 피크닉 가기 좋았던 요일은 '토요일'이었습니다", "작년 이맘때 전주는 평균 22도였습니다"와 같은 흥미로운 데이터 통계 카드를 제공합니다.
    

### 핵심 기능 2: 시간대별 기상 변화 기반 '반나절 코스' 자동 생성 (차별화 기능)

이 프로젝트의 꽃인 기능입니다. 날씨의 '변화'에 맞춰 장소의 '성격(실내/야외)'을 매칭합니다.

- **DB 사전 구축 흐름 (Place DB):** 전주 내 주요 스팟(덕진공원, 한옥마을, 세병호, 객리단길 카페, 전북대 앞 스터디카페 등)의 정보를 DB에 미리 저장합니다. 이때 가장 중요한 컬럼은 **`장소 유형 (야외/실내)`**과 **`위치 태그(덕진구/완산구)`**입니다.
    
- **사용자 요청 흐름:** 사용자가 "오늘 오후 1시부터 6시까지 전주 덕진구에서 놀 거야"라고 시간을 설정하고 버튼을 누릅니다.
    
- **백엔드 로직 흐름 (Data to LLM):**
    
    1. 백엔드 서버는 기상 예보 DB를 조회하여 1시~6시 사이의 시간대별 날씨 변화를 파악합니다. _(예: 1시~3시: 맑음, 온도 23도 / 4시~6시: 구름 많아짐, 바람 강해짐, 온도 18도로 하락)_
        
    2. 날씨 흐름에 맞춰 DB에서 장소를 1차 필터링합니다. _(조건: 1~3시는 야외 장소(덕진공원), 4~6시는 덕진공원 근처의 실내 장소(객사/전북대 근처 카페))_
        
    3. 추출된 날씨 데이터와 장소 데이터를 조합하여 **LLM(예: ChatGPT API 또는 Gemini API)에게 전달할 프롬프트(Context)를 생성**합니다.
        
- **결과 생성 및 렌더링 (AI RAG Flow):**
    
    LLM은 전달받은 데이터를 바탕으로 자연스러운 코스 설명 텍스트를 생성하여 프론트엔드로 반환합니다.
    
    > **[프론트엔드 출력 예시]**
    > 
    > 🕒 **오후 1:00 ~ 3:30 (따뜻한 야외 타임)**
    > 
    > "햇살이 가장 따뜻하고 미세먼지가 없는 시간대예요. **[덕진공원]**에서 돗자리를 펴고 샌드위치를 드시는 걸 추천해요!"
    > 
    > 🕒 **오후 4:00 ~ 6:00 (바람 피하기 타임)**
    > 
    > "4시부터는 서풍이 불면서 체감 온도가 떨어질 예정이에요. 야외 활동을 마무리하고, 공원 근처의 창이 넓은 실내 **[A 카페]**로 이동해서 따뜻한 차를 마시며 여유를 즐겨보세요."
    

## 4. 시스템 아키텍처 및 데이터 파이프라인 (Data Flow)

흐름을 중요하게 생각하시므로, 전체 시스템이 어떻게 맞물려 돌아가는지 정리했습니다.

1. **Data Collection (데이터 수집):** * `Spring Scheduler` 또는 `Node.js Cron`을 사용해 자정마다 기상청 API에서 과거 통계 데이터를 가져와 DB에 저장.
    
    - 실시간 예보 API는 1시간 단위로 호출하여 DB 업데이트 및 Redis 캐싱.
        
2. **Database (데이터 저장소):** * `Weather Table`: 날짜, 시간, 온도, 강수확률, 미세먼지 등.
    
    - `Place Table`: 장소명, 위치(좌표), 장소 유형(야외/실내), 이미지 URL.
        
3. **Backend Service (비즈니스 로직):**
    
    - 프론트엔드의 요청을 받아 DB에서 날씨와 장소를 Join하여 조회.
        
    - 조회된 데이터를 문장으로 엮어 외부 LLM API(OpenAI/Gemini)로 전송 후 응답 수신.
        
4. **Frontend Service (사용자 인터페이스):**
    
    - React 또는 Next.js를 활용하여, 시간의 흐름(타임라인 UI)이 잘 보이도록 예쁘게 렌더링. 애니메이션 효과(맑은 날씨엔 해 아이콘이 뜨는 등)를 추가하여 시각적 만족도 극대화.
        

## 5. 단계별 개발 순서 (Development Sequence)

데이터가 원활하게 흘러가려면 기반부터 탄탄히 다져야 합니다. 프론트엔드 화면을 먼저 그리기보다는, **'데이터 수집 -> 백엔드 처리 -> 화면 출력 -> AI 고도화'**라는 자연스러운 흐름에 따라 개발을 진행합니다.

### 1단계: 기획 구체화 및 DB 설계 (기반 공사)

- **목표:** 어떤 데이터를 어떻게 저장할지 뼈대를 잡습니다.
    
- **작업 내용:**
    
    - 날씨 정보(온도, 습도, 풍속 등)와 전주 장소 정보(야외/실내, 좌표 등)를 담을 RDBMS(MySQL 등) 테이블 스키마 설계.
        
    - 공공데이터포털(기상청) 및 에어코리아 API 키 발급 및 테스트.
        
    - 전주 주요 스팟 데이터 20~30개 정도를 DB에 초기 데이터(Seed Data)로 직접 입력.
        

### 2단계: 데이터 수집 파이프라인 구축 (데이터 채우기)

- **목표:** 빈 DB에 주기적으로 외부 데이터를 채워 넣는 자동화 흐름을 만듭니다.
    
- **작업 내용:**
    
    - 백엔드(Spring/Node.js)에서 스케줄러를 구현하여 매일/매시간 기상청 API를 호출해 내 DB에 적재하는 로직 개발.
        
    - 과거 통계 데이터를 스크립트로 한 번에 긁어와 DB에 밀어 넣는 배치(Batch) 작업 수행.
        
    - _(이 단계가 끝나면 내 DB만 조회해도 전주의 과거/현재 날씨를 알 수 있는 상태가 됩니다.)_
        

### 3단계: 기본 API 개발 및 프론트엔드 연동 (데이터 보여주기)

- **목표:** DB에 쌓인 데이터를 화면에 예쁘게 뿌려주는 기본 기능을 완성합니다.
    
- **작업 내용:**
    
    - 백엔드: 오늘 날씨 기반 '피크닉 지수'를 계산하는 수식 로직 개발 및 데이터 응답 API 생성. 과거 통계 데이터를 그룹화(Group by)하여 내려주는 API 생성.
        
    - 프론트엔드: 메인 페이지 UI 구현, 날씨에 따른 배경화면/아이콘 동적 렌더링, 통계 차트 화면 구현.
        

### 4단계: 시간대별 날씨-장소 매칭 로직 개발 (AI 적용 전 단계)

- **목표:** LLM을 붙이기 전에, 시스템 내부적으로 날씨와 장소를 연결하는 핵심 규칙(Rule)을 완성합니다.
    
- **작업 내용:**
    
    - 프론트엔드에서 사용자가 '시간대'를 선택할 수 있는 UI 추가.
        
    - 백엔드에서 해당 시간대의 날씨 변화(예: 비 옴 -> 맑음)를 조회하고, 그에 맞는 장소(실내 -> 야외)를 DB에서 SQL로 1차 필터링하여 매칭하는 로직 개발.
        
    - _이 시점에서는 AI의 문장이 아니라, 단순 리스트 형태로 "[1시~3시: A카페], [4시~6시: 덕진공원]"과 같이 결과가 잘 나오는지 테스트합니다._
        

### 5단계: RAG 기반 LLM 연동 (프로젝트 고도화 및 완성)

- **목표:** 앞서 만든 4단계의 딱딱한 매칭 결과를 LLM을 통해 사람의 언어로 부드럽게 변환합니다.
    
- **작업 내용:**
    
    - OpenAI 또는 Gemini API 연동.
        
    - 4단계에서 추출된 데이터(날씨 데이터 + 필터링된 장소 데이터)를 조합하여 LLM에 보낼 **시스템 프롬프트 템플릿** 작성. (예: "너는 전주 피크닉 가이드야. 다음 날씨와 장소 데이터를 보고 추천 코스를 문장으로 써줘. 데이터: ...")
        
    - 프론트엔드에 LLM이 생성한 자연스러운 추천 텍스트를 렌더링하는 UI(타임라인 말풍선 등) 추가.
        
    - 응답 속도 최적화 및 최종 디버깅.
        

## 6. 예상 기술 스택 (예시)

- **Frontend:** React.js (또는 Next.js), TailwindCSS (예쁜 UI/UX 구현)
    
- **Backend:** Java Spring Boot (또는 Node.js Express/NestJS)
    
- **Database:** MySQL (관계형 데이터 정리) + Redis (실시간 날씨 API 과부하 방지용 캐시)
    
- **External API:** 기상청 단기예보/과거데이터 API, 에어코리아(미세먼지) API, OpenAI/Gemini API (코스 설명 텍스트 생성용)
    

## 7. 기대 효과 및 학부 프로젝트로서의 의의

- 단순한 CRUD(생성/읽기/수정/삭제) 게시판을 넘어, 외부 데이터(API)를 내 DB로 끌어오고 가공하는 **데이터 파이프라인 구축 경험**을 어필할 수 있습니다.
    
- 정형화된 데이터(날씨 수치, 장소)를 LLM과 결합하여 비정형적이고 자연스러운 문장(코스 추천)으로 만들어내는 **최신 RAG(검색 증강 생성) 트렌드를 가볍게 경험**하고 포트폴리오에 녹일 수 있습니다.
