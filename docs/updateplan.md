
🧺 나들해 프로젝트: 피크닉 지수 시스템 및 API 아키텍처 설계서

이 문서는 '나들해' 웹 서비스의 핵심 기능인 피크닉 지수(점수) 산출 로직과 이를 뒷받침하는 공공데이터 API의 효율적 처리 방법을 정의합니다. 특히 사용자의 시각적 피로도를 줄이고, 위급 상황 시에만 정보를 표출하는 '선택적 노출(Event-Driven)' 철학을 반영했습니다.

1. 🎯 설계 철학: 데이터의 '상시 노출'과 '조건부 노출' 분리

모든 기상 데이터를 한 화면에 욱여넣으면 난해해집니다. 데이터를 성격에 따라 두 가지로 완벽히 분리합니다.

A. 상시 노출 데이터 (Routine Data)

특징: 피크닉을 계획할 때 항상 고려해야 하는 기본 척도입니다. 화면에 항상 예쁜 아이콘과 함께 표시됩니다.

항목: 기온(TMP), 하늘상태(SKY), 강수확률(POP), 미세먼지(PM10/25).

UI 표현: 메인 대시보드에 고정 표시.

B. 조건부 노출 데이터 (Event Data) - 🚨 핵심 변경점

특징: 평상시에는 데이터가 아예 화면에 렌더링되지 않습니다(숨김 처리). 특보나 재해가 발효된 '특정 이벤트 발생 시'에만 붉은색 배너나 경고 아이콘으로 강력하게 표출됩니다.

항목:

지진해일 통보 (getEqkMsg)

기상특보 (getWthrWrnList - 태풍, 폭설, 폭우, 강풍 주의보 등)

심각한 우천 (강수형태 PTY가 폭우/폭설일 때)

2. 🧮 피크닉 지수 2.0 산출 로직 (Knock-out 시스템)

기존의 단순 합산 방식에서 벗어나, 치명적인 요소가 발생하면 곧바로 점수를 '0점'으로 만들어버리는 녹아웃(Knock-out, 즉시 탈락) 필터를 최우선으로 적용합니다.

[1단계] 즉시 탈락 필터 (Priority Override)

이 단계에서 하나라도 걸리면 피크닉 지수는 즉시 0점(또는 최소점)으로 고정되며, 이후 계산을 생략합니다.

자연재해/특보 발효 여부: 지진 발생 API 응답이 있거나, 해당 지역(전주)에 기상특보(태풍, 호우, 대설 등)가 발효 중인가?

👉 YES: 지수 0점. 화면 최상단에 [🚨 기상특보 발효 중: 야외활동을 삼가세요] 배너 노출.

현재 강수 여부: 단기예보의 PTY(강수형태)가 1(비), 2(비/눈), 3(눈), 4(소나기) 중 하나인가?

👉 YES: 지수 10점. 화면에 우천 UI 표출. (SKY 등 다른 지표 계산 무의미)

3. 🌐 위치 기반 확장 및 API 한도 방어 전략 (Fallback Architecture)

전주 한정 서비스에서 '유저 위치 기반(Geolocation)' 서비스로 도약하기 위한 아키텍처입니다. 특히 가장 한도가 적은 에어코리아(일 500회)를 방어하는 것이 핵심입니다.

3.1. 캐싱 티어(Tier) 및 호출 전략

유저의 위경도를 그대로 API에 요청하지 않고, '행정구역/관측소 단위'로 그룹화하여 서버에 캐싱합니다.

Tier 1 (전주/전북대 - 최우선)

로직: 서버 스케줄러가 유저 접속 여부와 상관없이 매 1시간마다 무조건 API를 호출하여 DB/Redis에 갱신합니다.

비용: 하루 24회 호출 (절대 한도 초과 안 함).

Tier 2 (타 지역 - On-Demand 캐싱)

로직: 타 지역 유저가 접속하면 서버 캐시를 먼저 확인합니다. 해당 지역 데이터가 캐시에 없거나 1시간이 지났을 경우에만 API를 호출하고 결과를 캐싱합니다.

장점: 같은 지역(예: 서울 마포구)의 다른 유저가 접속하면 캐시된 데이터를 반환하여 API 호출을 아낍니다.

3.2. 한도 초과 시 최후의 보루 (Graceful Degradation)

타 지역 유저 폭주로 인해 에어코리아 API 일일 한도(500회)가 모두 소진되었을 때, 서비스가 다운되지 않도록 하는 우회(Fallback) 시스템입니다.

서버가 타 지역 API 호출 시 Limit Exceeded (에러 코드)를 감지합니다.

에러 발생 시 프론트엔드로 fallback: true 플래그를 담은 응답을 보냅니다. 이때 데이터는 Tier 1에서 안전하게 확보해둔 '전주' 데이터를 강제로 내려줍니다.

프론트엔드는 이 플래그를 확인하고 화면에 토스트 알림을 띄웁니다.

UI 메시지: "현재 계신 지역의 대기 정보를 일시적으로 불러올 수 없어, 나들해의 홈타운인 전주 기준 날씨를 보여드려요! 🏡"

4. 💬 지역별 맞춤 피크닉 브리핑 시스템

위치 정보가 확보됨에 따라, 메인 화면의 피크닉 브리핑(멘트)을 지역에 맞춰 이원화합니다. 전주는 나들해의 본거지이므로 특별한 로컬 감성을 부여합니다.

A. 전주 지역 (Priority Location) 브리핑 예시

전주 데이터가 표출될 때(유저 위치가 전주이거나, Fallback이 발동했을 때)는 로컬 명소를 적극 활용합니다.

[지수 80 이상]: "바람도 온도도 완벽해요! 덕진공원에 돗자리 펴기 딱 좋은 날이네요. 전북대 앞 카페에서 커피 한 잔 테이크아웃 어떠세요? ☕️"

[지수 60~79]: "나들이하기 무난한 날씨예요. 객리단길을 산책하거나 건지산으로 가벼운 피크닉을 떠나보세요. 🍃"

[지수 40 미만]: "오늘은 밖에서 오래 놀기엔 아쉬운 날씨예요. 전북대 구정문 쪽 실내 아지트를 찾아볼까요? ☔️"

B. 타 지역 (Generic Location) 브리핑 예시

그 외 지역 유저에게는 현재 날씨 상태에 맞춘 보편적이고 감성적인 멘트를 제공합니다.

[지수 80 이상]: "햇살이 눈부신 완벽한 날씨! 당장 근처 공원으로 돗자리 하나 들고 떠나보세요. 🧺"

[지수 60~79]: "가볍게 산책하기 좋은 날씨네요. 근처 테라스 카페에서 여유를 즐겨보는 건 어떨까요? 🌿"

[지수 40 미만]: "날씨가 조금 변덕스럽네요. 오늘은 포근한 실내에서 맛있는 걸 먹으며 쉬어가는 걸 추천해요. 🛋️"
